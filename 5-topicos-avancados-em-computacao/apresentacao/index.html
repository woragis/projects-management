O problema

A desinformação se espalha mais rápido que a verificação. O usuário comum não tem tempo nem ferramentas para checar cada manchete.

📱

Volume absurdo

Milhares de links e textos circulam a cada hora em redes, grupos e portais.

🤔

Difícil de avaliar

Manchetes sensacionalistas parecem plausíveis; fontes reais exigem cruzamento de dados.

🔁

Repetição sem checagem

A mesma notícia é compartilhada dezenas de vezes sem ninguém confirmar o conteúdo.

“Como saber se posso confiar nessa matéria agora, sem virar fact-checker?”

Arquitetura macro

Microsserviços orquestrados via Docker Compose — cada peça com responsabilidade clara.

  • Frontend — portal, dashboard e detalhe da análise
  • API — autenticação, cache, regras do portal
  • IA — scrape, claims, fact-check, score
  • PostgreSQL — catálogo público e histórico de versões
  • RabbitMQ — logging sem bloquear a resposta ao usuário

Dois cenários de fluxo

O comportamento muda conforme a URL já foi analisada — economia de IA e transparência para o usuário.

Cenário A

URL nova

  1. POST /api/analyses
  2. Cache miss no PostgreSQL
  3. IA raspa, extrai claims, consulta Serper
  4. LLM calcula score e veredito
  5. Salva no banco + log na fila
  6. Entra no portal público
85%
Confiável

Cenário B

URL em cache

  1. POST /api/analyses
  2. Cache hit — IA não é chamada
  3. Serper e scrape pulados
  4. Devolve análise existente
  5. Banner: “análise anterior em …”
  6. Reanálise só após cooldown 24h
⏱
Resultado reutilizado

Resposta rápida, sem custo de IA

// backend · regras de negócio

Cache e políticas do portal

Decisões que evitam retrabalho na IA e definem o comportamento público do catálogo.

🔑

Chave canônica

URL normalizada vira canonicalKey — mesmo link com parâmetros diferentes reutiliza o resultado.

URL + texto no envio → só a URL define a chave.

⏳

Cooldown 24h

Reanálise bloqueada até REANALYZE_COOLDOWN_HOURS=24 após a última rodada.

API retorna canReanalyze e nextReanalyzeAt.

🗑

Janela de exclusão

Quem submeteu primeiro pode excluir em até 30 minutos (DELETE_WINDOW_MINUTES).

Remove ContentItem e análises vinculadas.

📖

Leitura pública

Portal acessível sem JWT — qualquer visitante lê feed e detalhe.

Escrita (escanear, reanalisar) exige login.

// serviço de IA · FastAPI :8000

Pipeline de análise

Endpoint POST /ai/analyze — pipeline sequencial com tratamento de erro em cada etapa.

1
Scrape da URL

Extrai corpo do artigo (G1, Globo…). Com URL, o texto opcional do usuário é ignorado.

2
Extração de claims

LLM + spaCy identificam alegações factuais verificáveis no texto.

3
Fact-check

Google Fact Check Tools + fallback Serper para corroborar cada claim.

4
Score de credibilidade

LLM pondera claims, reputação do domínio, data de referência e fontes.

5
Resposta estruturada

credibility_score, verdict, claims[], sources[], explanation em PT.

// aprendizados

Desafios e soluções

Problemas reais encontrados durante o desenvolvimento e como foram resolvidos.

Corrigido

URL + texto ignorava o artigo

IA analisava a pergunta do usuário em vez de raspar a URL. Fix: URL sempre dispara scrape.

Corrigido

Fact-check vazio = falso

Inconclusivo era tratado como negativo. Fix: scorer pondera inconclusivo separadamente + Serper.

Corrigido

Migração Prisma

db push falhou com dados legados. Fix: migração SQL manual + migrate deploy.

Limitação

Dependência de APIs externas

Custo e latência do LLM; scrape falha em sites com paywall ou layout atípico.
